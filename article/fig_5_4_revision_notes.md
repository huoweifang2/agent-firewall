# 图 5.6 实验段落与图片修订说明

## 1. 当前问题

`article/images/fig_5_4_failure_distribution.png` 当前采用 `fast` 与 `balanced` 漏报类别并排柱状图。该图的问题不是单纯美观问题，而是实验叙事问题：两种策略在当前仓库复算口径下只相差 4 个场景，旧正文中使用的 `269/358`、`274/358` 和 `84 个漏报` 也已经不能代表当前代码状态。因此，继续用该图作为主要漏报分析证据，会让读者感觉 `fast` 与 `balanced` 没有实质区别，也会削弱实验章节的说服力。

建议将该图从“fast vs balanced 漏报类别对比”改为“策略阶梯与剩余能力边界”。也就是说，图中重点不再是两个相近策略之间的微小差异，而是展示 direct、fast、balanced、strict、paranoid 五种安全边界下的整体趋势，以及最高强度策略仍未覆盖的攻击类别。

## 2. 当前复算参考值

以下数值来自当前仓库对 358 个 JSON 项目回归场景的只读复算，可作为后续正式重写和重画图时的基准：

| 策略 | 正确场景数 | 场景总数 | 剩余未符合预期 |
| --- | ---: | ---: | ---: |
| direct | 20 | 358 | 338 |
| fast | 327 | 358 | 31 |
| balanced | 331 | 358 | 27 |
| strict | 331 | 358 | 27 |
| paranoid | 344 | 358 | 14 |

策略差异上，`fast -> balanced` 修复 4 个场景，`balanced -> strict` 没有额外修复，`strict -> paranoid` 修复 13 个场景。这个结果说明，默认 `balanced` 已明显优于未保护 direct 边界，但 `strict` 在当前场景和当前实现下没有带来可观增益；真正拉开差异的是更低阈值和更强安全配置的 `paranoid`。

如果采用这组数值，全文不能继续写 `fast=269/358`、`balanced=274/358` 或 `balanced` 剩余 `84` 个漏报，应同步更新摘要、第 5 章结果表、图注和附录复算说明。

### 4.2 第 5.7.1 建议正文

```markdown
为了更清楚地区分不同安全边界的能力，本文将原有 direct、fast 和 balanced 对比扩展为策略阶梯实验。direct 表示未保护 LLM 直连边界，不执行扫描、审计和工具门控；fast 只启用轻量规则和较高风险阈值；balanced 是系统默认策略；strict 加入更完整的扫描节点和更低阈值；paranoid 进一步降低风险阈值并启用更强安全配置。

在 358 个 JSON 项目回归场景上，direct 只能正确处理 20 个安全 ALLOW 样本；fast、balanced、strict 和 paranoid 分别达到 327/358、331/358、331/358 和 344/358。该结果说明，加入确定性安全边界后，系统相对未保护直连有明显提升；但 balanced 到 strict 在当前场景上没有产生额外增益，说明单纯增加策略节点并不必然改善检测效果。paranoid 通过更低风险阈值进一步减少剩余漏报，但该结果仍应理解为当前项目回归场景下的策略表现，而不是公开 benchmark 的通用安全分数。
```

### 4.3 第 5.7.1 建议表格

```markdown
| 方法 | 安全边界说明 | 正确场景数 | 场景总数 | 剩余未符合预期 |
| --- | --- | ---: | ---: | ---: |
| 未保护直连 LLM | `/v1/chat/direct`，不扫描、不审计、不门控 | 20 | 358 | 338 |
| `fast` rules-only | 最小规则与较高风险阈值 | 327 | 358 | 31 |
| `balanced` 默认策略 | rules + scanner nodes + risk aggregation | 331 | 358 | 27 |
| `strict` 严格策略 | 更完整扫描节点与更低风险阈值 | 331 | 358 | 27 |
| `paranoid` 高敏策略 | 更强安全配置与最低风险阈值 | 344 | 358 | 14 |
```

### 4.4 第 5.7.5 建议正文

```markdown
策略阶梯实验表明，`balanced` 相比 `fast` 只额外修复 4 个场景，`strict` 相比 `balanced` 没有新增修复，而 `paranoid` 进一步修复 13 个场景。因此，漏报分析的重点不应放在 `fast` 与 `balanced` 的并列对比，而应放在剩余能力边界上。当前 `balanced` 仍有 27 个攻击或敏感样本未符合预期，`paranoid` 仍有 14 个样本未符合预期，主要集中在凭据变体、编码混淆、Agent 语境下的 obfuscation、RAG poisoning、Prompt Injection (Agent) 和 Advanced Multi-Turn 等类别。

这些失败说明，Agent-Firewall 的架构能够提供模型前扫描、工具前门控、工具后清洗和 Trace 证据链，但检测信号本身仍需要增强。尤其是自然语言描述的密钥、非标准凭据格式、跨语言攻击、分片载荷、长期记忆污染和多轮渐进诱导，不能只依赖单轮规则与风险阈值完成判定。后续应将这些类别作为高优先级回归集合，而不是简单把它们归因于测试环境问题。
```

## 5. 建议新图设计

建议仍覆盖原文件 `article/images/fig_5_4_failure_distribution.png`，避免正文图片路径大范围改动。新图使用 SciencePlots 风格，做成左右双面板：

1. 左面板：策略阶梯柱状图。
   - 横轴：direct、fast、balanced、strict、paranoid。
   - 纵轴：正确场景数，或剩余未符合预期场景数。
   - 推荐画“剩余未符合预期”，因为能直接突出安全边界差异：338、31、27、27、14。
   - 在柱子上标注 `20/358`、`327/358`、`331/358`、`331/358`、`344/358`。

2. 右面板：剩余漏报类别 Top-N。
   - 推荐优先展示 `paranoid` 剩余 14 个场景的类别，因为它代表当前最强策略之后仍无法覆盖的能力边界。
   - 若论文希望重点讨论默认配置，则展示 `balanced` 剩余 27 个场景，并在正文说明 `paranoid` 进一步降低但不能完全消除漏报。

3. 图标题建议：
   - `策略强度与剩余漏报类别分布`
   - 英文副标题可不加，避免本科论文图面过载。

4. 不建议继续画 `fast` 与 `balanced` 并排类别柱状图，因为该图视觉差异过小，读者难以从图中获得新的实验信息。

## 6. 写作边界

这组实验结果应表述为“当前 358 个 JSON 项目回归场景下的策略阶梯结果”，不能写成 AgentDojo、ToolEmu 或 Agent Security Bench 官方分数。`paranoid=344/358` 也不能被解释为系统已经具备通用生产级安全能力，只能说明在当前项目场景库中，更严格阈值减少了部分漏报。

如果后续正式采用这组新数值，需同步更新中英文摘要、正文第 5 章、结论局限性和附录 A.6。否则，正文、图和附录之间会出现实验口径不一致。
