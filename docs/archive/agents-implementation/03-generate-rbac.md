# Step 3 — Generate RBAC

**Time: 10 minutes**
**Input: tool registry from Step 2**
**Output: `rbac.yaml` with roles, inheritance, and default-deny**

---

## Why this step matters

RBAC is the single most impactful security control for agents.
Without it, every user can call every tool. With it, a customer
can search the FAQ but can't issue refunds or access admin secrets.

You don't write RBAC from scratch. Agent-Firewall generates it from
your tool registry and risk profile, then you review and adjust.

---

## How RBAC generation works

Agent-Firewall looks at your tools and applies these rules:

| Tool property | RBAC effect |
|--------------|-------------|
| `sensitivity: low` | Available to all roles |
| `sensitivity: medium` | Available to `support` and `admin` |
| `sensitivity: high` | Available to `admin` only |
| `sensitivity: critical` | Available to `admin` + `requires_confirmation` |
| `access: write` | `requires_confirmation: true` for non-admin |
| `returns_pii: true` | Post-tool PII redaction for `customer` role |

### Default role hierarchy

Agent-Firewall generates 3 roles by default (you can rename or add more):

```
customer   →   lowest privileges, read-only, external users
    ↑
support    →   inherits customer + medium-sensitivity tools
    ↑
admin      →   inherits support + high/critical + write tools
```

Inheritance means: `admin` automatically gets everything `support` has,
and `support` gets everything `customer` has. You only define what's
**new** at each level.

## Generated output: rbac.yaml

Based on the Customer Support Copilot tool registry from Step 2:

```yaml
# rbac.yaml — generated by Agent-Firewall
# Agent: Customer Support Copilot
# Generated: 2026-03-09
#
# Rule: default-deny. If (role, tool) is not listed → BLOCKED.
# Inheritance: child role inherits all tools from parent.

roles:
  customer:
    description: "End-user customer — read-only access to public tools"
    tools:
      searchKnowledgeBase:
        scopes: [read]
        sensitivity: low
      getOrderStatus:
        scopes: [read]
        sensitivity: low

  support:
    description: "Support agent — customer tools + profile access"
    inherits: customer
    tools:
      getCustomerProfile:
        scopes: [read]
        sensitivity: medium
        # Note: returns PII — post-tool gate will redact for non-support roles

  admin:
    description: "System administrator — full access, write actions with confirmation"
    inherits: support
    tools:
      getInternalSecrets:
        scopes: [read]
        sensitivity: critical
        requires_confirmation: true
      issueRefund:
        scopes: [write]
        sensitivity: high
        requires_confirmation: true
```

### What this means in practice

| Action | customer | support | admin |
|--------|----------|---------|-------|
| Search FAQ | ✅ | ✅ | ✅ |
| Check order status | ✅ | ✅ | ✅ |
| View customer profile | ❌ BLOCKED | ✅ | ✅ |
| Issue refund | ❌ BLOCKED | ❌ BLOCKED | ✅ (confirm) |
| Access secrets | ❌ BLOCKED | ❌ BLOCKED | ✅ (confirm) |

## Customization

The generated RBAC is a starting point. Common adjustments:

### Add a new role

```yaml
roles:
  supervisor:
    description: "Team lead — can issue refunds without confirmation"
    inherits: support
    tools:
      issueRefund:
        scopes: [write]
        sensitivity: high
        requires_confirmation: false  # supervisors skip confirmation
```

### Add scope restrictions

```yaml
  support:
    tools:
      getCustomerProfile:
        scopes: [read]          # can read but NOT update
        sensitivity: medium
        conditions:
          - own_customers_only   # optional: scope to assigned customers
```

### Remove confirmation for trusted roles

```yaml
  admin:
    tools:
      issueRefund:
        requires_confirmation: false  # admins skip confirmation
```

### Add rate limits per role

```yaml
  customer:
    tools:
      searchKnowledgeBase:
        scopes: [read]
        sensitivity: low
        rate_limit: 10           # max 10 calls per session
```

---

## Review checklist

Before moving on, verify:

- [ ] **Default-deny works** — roles not listed for a tool can't use it
- [ ] **Write tools require confirmation** — at least for customer/support
- [ ] **Critical tools are admin-only** — or at minimum require confirmation
- [ ] **Inheritance makes sense** — each child role only adds, never removes
- [ ] **PII tools have post-scanning** — `returns_pii: true` in tools.yaml
- [ ] **Rate limits are set** — especially for customer-facing tools
- [ ] **No role has access to ALL tools** — least privilege principle

---

## Validation

Agent-Firewall checks your RBAC config for common mistakes:

| Check | Status |
|-------|--------|
| Every tool from registry has at least one role mapped | ✅ / ⚠️ |
| No `customer` role has access to `write` tools | ✅ / ⚠️ |
| All `critical` tools require confirmation | ✅ / ⚠️ |
| Inheritance chain is acyclic | ✅ / ⚠️ |
| No overlapping permission conflicts | ✅ / ⚠️ |
| At least one role exists | ✅ / ⚠️ |
